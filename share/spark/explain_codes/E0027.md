Example of illegal code:

```ada
with Ada.Unchecked_Conversion;
with Interfaces; use Interfaces;

procedure Example with SPARK_Mode => On is
   type Int_Access is access all Integer;

   function Conv is new Ada.Unchecked_Conversion
     (Int_Access, Unsigned_64);  -- error
begin
   null;
end Example;
```

This error is issued on an instantiation of `Ada.Unchecked_Conversion` whose
source type is an access type or a composite type containing access
subcomponents. As stated in [section 13.9(1) of the SPARK Reference Manual],
such conversions are not allowed in SPARK.

The restriction exists because `gnatprove` does not model the raw bit
representation of access values. An access value's internal representation is
implementation-defined, and reinterpreting those bits as another type produces
a result whose meaning cannot be reasoned about formally. Additionally, such
conversions bypass SPARK's ownership model: converting an access value to an
integer and back could create aliases that are invisible to the ownership
analysis.

The simplest fix is to avoid types containing access subcomponents as the
source of `Unchecked_Conversion`. If the conversion is required for low-level
interfacing purposes, encapsulate it in a subprogram or package whose
`SPARK_Mode` is set to `Off`:

```ada
with Ada.Unchecked_Conversion;
with Interfaces; use Interfaces;

package Low_Level
  with SPARK_Mode => Off
is
   type Int_Access is access all Integer;

   function Conv is new Ada.Unchecked_Conversion
     (Int_Access, Unsigned_64);
end Low_Level;
```

Client code that does not need to use the conversion directly can still be
analyzed in SPARK. The programmer must then ensure that the excluded code does
not violate any assumptions made in the SPARK-analyzed parts.

[section 13.9(1) of the SPARK Reference Manual]:
https://docs.adacore.com/live/wave/spark2014/html/spark2014_rm/representation-issues.html#unchecked-type-conversions
