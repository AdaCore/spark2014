Example of illegal code:

```ada
type T is tagged record
   Value : Integer;
end record;

procedure Op (X : T)
  with Pre => X.Value > 0;  -- error
```

This error is issued on primitive operations of a tagged type that carry a
`Pre` aspect. As stated in [section 6.1.1(2) of the SPARK Reference Manual],
plain preconditions are not allowed on dispatching subprograms; `Pre'Class`
must be used instead.

The reason is that `Pre` and dynamic dispatch interact in a way that makes
verification unsound or redundant. When a dispatching call is made, the
subprogram that actually executes is determined at runtime by the tag of the
controlling operand. A caller has only static knowledge of the declared type
and can only check the base subprogram's `Pre`. For verification to remain
sound, SPARK would have to require every override to carry an identical
`Pre'Class` anyway — making the use of `Pre` both redundant and confusing.
SPARK therefore requires `Pre'Class` directly.

`Pre'Class` expresses a class-wide precondition that governs all overrides: an
override may only *weaken* the precondition (for example by adding an `or else`
disjunct), consistent with the Liskov Substitution Principle. Callers
dispatching through the base type's interface can then rely on the class-wide
contract regardless of which override executes at runtime.

The fix is to replace `Pre` with `Pre'Class`:

```ada
procedure Op (X : T)
  with Pre'Class => X.Value > 0;  --  OK
```

[section 6.1.1(2) of the SPARK Reference Manual]:
https://docs.adacore.com/live/wave/spark2014/html/spark2014_rm/subprograms.html#preconditions-and-postconditions
