Example of illegal code:

```ada
X : Integer with Ghost, Volatile; -- error
```

This error is issued on the declaration of a ghost object or type that is also
volatile (or effectively volatile). It violates legality rule 9 of
[section 6.9 of SPARK Reference Manual].

Ghost entities exist solely for verification purposes and must have no effect on
the dynamic behavior of a program. Volatile objects, by contrast, are explicitly
intended to have externally visible side effects — their reads and writes are
observable by hardware, other tasks, or external agents. Allowing a ghost object
to be volatile would be contradictory: the object is supposed to be erasable
without changing program semantics, yet volatile semantics require that its
accesses be preserved. SPARK therefore forbids combining the two properties.

The fix is to remove the `Volatile` (or `Async_Readers`, `Async_Writers`,
`Effective_Reads`, or `Effective_Writes`) aspect from the ghost declaration:

```ada
X : Integer with Ghost; -- ok
```

If the intent was to model an external volatile memory location for verification
purposes, the recommended approach is to keep the volatile object non-ghost and
expose its effects through a package with an external abstract state. The
volatile variable becomes a non-ghost constituent in the body, and callers are
informed of its participation through `Global` contracts:

```ada
package Sensor
  with SPARK_Mode => On,
       Abstract_State => (State with External => Async_Writers)
is
   procedure Read (V : out Integer)
     with Global => (Input => State);
end Sensor;
```

```ada
package body Sensor
  with SPARK_Mode => On,
       Refined_State => (State => Raw)
is
   Raw : Integer with Volatile, Async_Writers;

   procedure Read (V : out Integer)
     with Refined_Global => (Input => Raw)
   is
   begin
      V := Raw;
   end Read;

end Sensor;
```

The `Abstract_State` with `External => Async_Writers` tells GNATprove that the
package has external volatile state that external agents may write asynchronously.
The body refines `State` to the concrete volatile variable `Raw`, and the
`Global` contract on `Read` documents that each call reads from that external
state. Because the volatile value may change between any two Ada statements, it
is not possible to write a postcondition relating the output to the current value
of `Raw` — the `Global` contract is the appropriate verification interface here.

For more information, see [section 6.9 of the SPARK Reference Manual].
