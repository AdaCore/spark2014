Example of illegal code:

```ada
package P with SPARK_Mode => On is

   type Element_Type is tagged null record;

   type Container is (Holder)
     with Iterable => (First       => First,
                       Next        => Next,
                       Has_Element => Has_Element,
                       Element     => Element);  -- error: Element is a primitive
                       --        of the tagged type Element_Type and so has a
                       --        controlling result

   function First       (X : Container) return Boolean is (True);
   function Next        (X : Container; Y : Boolean) return Boolean is (False);
   function Has_Element (X : Container; Y : Boolean) return Boolean is (Y);
   function Element     (X : Container; Y : Boolean) return Element_Type
   is (null record);

end P;
```

This error is issued on a function named in the `Iterable` aspect of a
container type (i.e., one of `First`, `Next`, `Last`, `Previous`,
`Has_Element`, `Element`, or `Constant_Reference`) when that function has a
controlling result. A function has a controlling result when its result type
is a specific tagged type and the function is a primitive of that type, which
makes the function dispatching on its result.

[Section 5.5.2(12) of the SPARK Reference Manual] forbids this because the
generalized iteration scheme defined by the `Iterable` aspect calls these
functions statically; a dispatching call on the result would require the
caller to supply a controlling tag, which the iteration machinery does not
provide. Allowing a controlling result would also leave the dynamic type of
the returned cursor or element under-specified, which `gnatprove` cannot
reason about.

The fix is to ensure that none of the functions named in the `Iterable`
aspect dispatches on its result.

The most common approach, and the one used by the iterable container types
in `SPARK.Containers.Formal.*`, is to declare the container type and its
`Iterable` primitives in a different package from the one declaring the
tagged result type. A function is a primitive of a tagged type only when it
is declared in the same package as that type, so moving the iteration
functions to a separate package (here, a child package) removes the
controlling result:

```ada
package P with SPARK_Mode => On is

   type Element_Type is tagged null record;

end P;

package P.Iter with SPARK_Mode => On is

   type Container is (Holder)
     with Iterable => (First       => First,
                       Next        => Next,
                       Has_Element => Has_Element,
                       Element     => Element);

   function First       (X : Container) return Boolean is (True);
   function Next        (X : Container; Y : Boolean) return Boolean is (False);
   function Has_Element (X : Container; Y : Boolean) return Boolean is (Y);
   function Element     (X : Container; Y : Boolean) return P.Element_Type
   is (P.Element_Type'(null record));

end P.Iter;
```

Alternatively, change the result type of the offending function to the
class-wide type `Element_Type'Class`. A class-wide result is not a
controlling result (per Ada RM 3.9.2), so the function is no longer a
dispatching operation, even though it remains a primitive of the tagged
type. This keeps the `Iterable` declaration in the same package as the
tagged type, at the cost of making the iteration yield class-wide values:

```ada
package P with SPARK_Mode => On is

   type Element_Type is tagged null record;

   type Container is (Holder)
     with Iterable => (First       => First,
                       Next        => Next,
                       Has_Element => Has_Element,
                       Element     => Element);

   function First       (X : Container) return Boolean is (True);
   function Next        (X : Container; Y : Boolean) return Boolean is (False);
   function Has_Element (X : Container; Y : Boolean) return Boolean is (Y);
   function Element     (X : Container; Y : Boolean) return Element_Type'Class
   is (Element_Type'(null record));

end P;
```
